## Design Decisions
- **Agent skill, not CLI command**: execute-plan must invoke other skills (`/tr-behaviour`, `/tr-implement`, `/tr-refine`) and update `taproot/plan.md` after each outcome — these require agent reasoning and skill delegation. The implementation is a skill file.
- **Mode detection from natural language**: four modes (step-by-step, batch, specify, implement) are inferred from the developer's request via a lookup of common phrasings, with step-by-step as default. No explicit `--mode` flag needed.
- **Filters compose with step-by-step/batch**: specify and implement modes filter the pending item list before the main execution loop; the loop then runs in step-by-step or batch as usual. This keeps the flow simple.
- **Filtered-out items stay `pending`**: `skipped` implies developer intent to bypass; filtered items should be processed in a later pass. This enables the two-pass specify→implement workflow.
- **Per-item presentation preserved in batch**: the spec is explicit that batch grants up-front confirmation but still shows each item as it executes. The agent presents then proceeds without waiting.
- **Skill slug `plan-execute`**: groups build/execute/analyse under the `plan-` namespace.
